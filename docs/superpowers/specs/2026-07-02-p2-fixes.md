## P2-4: Agent 上下文管理隐患

### 问题

1. `max_tokens` 硬编码 128000 — 只适配 GPT-4o/DeepSeek-V4，换 32K 或 200K 模型就出错
2. 裁剪在循环开始、追加在循环末尾 → 大型 tool result 追加后累积到下一轮才裁

### 方案

**问题 1 — 自适应上下文窗口 + JSON 查表加速**：

API 不返回上下文窗口信息（DeepSeek `/v1/models` 实测确认）。

方案：**JSON 只是优化起点，自适应二分是兜底**。

```
初始值 = .env LLM_MAX_CONTEXT (>0)
         || model_contexts.json 命中值
         || 128000

调用 LLM：
  ├─ 成功 → 记住这个值，会话内复用                 ← 快路径
  └─ HTTP 400 + context_length 相关错误？
        ├─ 是 → 砍半重试，最低 16000               ← 自适应
        └─ 否 → 正常抛异常
```

为什么实测可行：

| provider 行为 | 自适应效果 |
|-------------|-----------|
| 严格报错（OpenAI/Moonshot/GLM） | 捕获 → 砍半 → 自动适配 ✓ |
| 静默截断（DeepSeek） | 不报错 → 探测通过 ✓ |

DeepSeek 传 150K tokens 直接成功不报错（实测），说明静默截断的 provider 也不会触发错误循环。

`model_contexts.json` 覆盖约 30 个热门模型（前缀匹配），让绝大多数情况走快路径（一次过），未知模型才走自适应探测。

**`agent/context_window.py` 新建**，封装上述逻辑：
```python
async def detect_context_window(llm, initial: int) -> int:
    """自适应二分探测模型上下文窗口上限。"""
    budget = initial
    while budget >= 16000:
        try:
            await _probe(llm, budget)
            return budget
        except _ContextLengthError:
            budget //= 2
    raise ContextDetectionError(
        f"无法适配当前模型 {settings.llm_model}，上下文窗口探测至 16000 仍然失败，"
        "请更换模型或在 .env 中手动设置 LLM_MAX_CONTEXT"
    )
```

异常通过 SSE error 事件传递到前端 → 错误栏显示 → 用户看到提示并更换模型。（P2-3 修复后错误栏已可用）

**问题 2 — 追加后立即裁剪**：
```python
# loop.py — 所有 tool result 追加完毕后
messages, _, _ = ctx_manager.trim_messages(messages)
iteration += 1
continue
```
