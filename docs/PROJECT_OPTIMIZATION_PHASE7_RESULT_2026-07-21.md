# 项目优化第七阶段结果（2026-07-21）

## 结论

本阶段针对真实多轮问答中“应比较 Skill 与 MCP，却只返回 MCP 一句定义”的同类风险进行了代码级审计和系统性修复。修复不只覆盖原始问法，也覆盖 `vs`、`相比`、`相较于`、`怎么选`、`它们`、`前者/后者`、职责、原因、影响、关系和计算类问题。

Agent 全量测试 298/298 通过；93×2 历史在线答案经当前代码零模型调用重评分后，质量门禁和独立发布门禁均通过。本阶段未调用 Claude，未向外部模型 API 发送新请求。

## 1. 实际风险复现与审计结果

审计确认原问题不是单一提示词问题，而是以下路径可能共同导致答案过短或答非所问：

1. 比较检测词覆盖不全，`MCP vs Skill`、`相比`、`怎么选` 等问法可能进入逐句流式放行；
2. `它们`、`前者`、`后者` 和“详细说说”等追问缺少稳定的多轮解析；
3. 旧轮工具消息和旧引用编号可能继续进入当前上下文，与当前验证器持有的来源不一致；
4. 来源数量未发生裁剪时，旧检索消息不会被清空；
5. 关系类问题可能只返回相关定义，却没有回答职责、原因、影响或计算关系；
6. 修复答案按忠实度优先选择时，可能把多个已支持事实压缩成单条事实；
7. 生产路径关闭覆盖度复查，概览和定义类答案可能在证据充足时仍然过短。

## 2. 已实施优化

### 2.1 统一查询语义层

新增 `backend/agent/query_semantics.py`，集中实现：

- 比较题和关系敏感题识别；
- 显式比较双方提取；
- 单数、复数及前后项指代解析；
- 简短追问继承上一轮明确主题；
- 历史工具消息剔除、历史回答引用编号清理；
- 判断哪些问题必须等待完整答案后再验证。

无法安全解析的指代保持原样并触发明确追问，不再产生 `Skill们` 一类错误替换。

### 2.2 当前轮证据隔离

进入 Agent Loop 前只保留历史用户与助手文本，移除旧工具消息、旧工具调用及旧 `[S#]` 引用。缓存键同样使用清理后的历史，避免不同轮次的临时引用污染缓存身份。

来源归一化不再依赖“是否真的裁剪了来源数量”：每轮都会清空旧搜索工具消息，只在最新搜索消息中写入与当前验证器完全一致的来源集合。

### 2.3 完整答案验证

比较、指代、职责、因果、影响、关系、计算和最高级问题不再逐句放行，而是在完整生成后统一验证。比较答案必须：

- 明确表达比较关系；
- 在查询可提取双方名称时，同时覆盖双方；
- 不能仅凭单侧句子中的“适合”通过完整性守卫。

职责、原因、影响和关系问题增加确定性关系守卫。例如，“Django 使用 MTV 架构”不能再冒充对“MTV 每层职责是什么”的完整回答。

### 2.4 防止修复结果信息坍缩

当原答案含至少两个有证据支持的事实，而修复答案减少了支持事实数量时，系统会保留原答案中所有确定支持的声明及其完整引用集合，并再次验证忠实度、引用召回率和引用精确率。只有验证全部为 100% 时才采用该保守重建结果。

### 2.5 恢复覆盖度复查

生产路径默认启用 `grounding_coverage_recheck_enabled`。定义、概览和详细说明类问题在证据内容明显丰富而答案过短时，会进入既有覆盖度复查流程。该开关仍可通过配置关闭。

缓存提示词版本已提升为 `v6-relation-complete`，避免复用修复前的单侧或关系缺失答案。

## 3. 回归测试

新增或扩展的回归用例覆盖：

- `MCP 和它有什么区别`；
- `它们各适合什么场景`、`前者/后者有什么优势`；
- `MCP vs Skill`、`MCP相比Skill`、`MCP和Skill怎么选`；
- 无可解析对象时拒绝错误替换；
- 职责、因果、影响、关系和计算题强制整答验证；
- 即使来源数量没有减少，也清除旧轮搜索证据；
- 历史工具消息、工具调用和旧引用不进入新轮；
- 修复不能把两个已支持事实压缩成一个。

## 5. 剩余边界

本轮已经封堵已发现的确定性同类路径，但无法承诺任何生成模型在所有未知表达下永不产生短答。当前保障方式是：常见变体在生成前被识别，关系敏感回答在生成后被完整检查，无法确定时明确拒答或要求补充对象，并由回归矩阵防止未来代码改动重新引入这些问题。

固定管理员密码策略继续按用户要求暂缓，本阶段未改动。
