# 文档入库生命周期加固方案

> 日期：2026-07-16  
> 状态：已执行

## 1. 优化目标

解决上传、后台入库、列表刷新、删除、清空和服务重启之间的竞态，使文档生命周期最终必然收敛到 `ready`、`failed` 或已删除，避免永久停留在“上传完成，等待处理”。

## 2. 已识别风险

1. 上传期间发起的旧列表请求可能在上传响应之后返回，用空快照覆盖新文档。
2. 服务端已经删除文档时，前端仍永久保留 `activeUploadDocId`，形成幽灵行和上传锁。
3. 清空/删除正在处理的文档不会停止后台任务，任务会继续访问已删除记录和文件。
4. 服务重启后，任务队列表只恢复为 `pending`，没有消费者真正重放入库任务。
5. 前端声明多文件上传，但全局状态实际上只能处理一个文件。

## 3. 实施方案

### 3.1 前端状态一致性

- 每次列表请求记录请求序号和开始时的本地变更版本。
- 不是最新请求，或请求期间发生上传/SSE/重试变更时，丢弃该响应。
- 最新服务端响应确认活动文档缺失时，删除幽灵行并释放上传锁。
- 清空操作集中到 Store，统一关闭进度流、取消请求并重置状态。

### 3.2 删除与清空保护

- 后端拒绝删除非 `ready/failed` 文档，返回 HTTP 409。
- 后端存在非终态文档时拒绝清空全部。
- 前端存在处理中任务时禁用“清空全部”，并显示原因。

### 3.3 服务重启恢复

- 启动时扫描 `uploaded/parsing/chunking/embedding/indexing` 文档。
- 原文件存在：状态重置为 `uploaded` 并重新调度完整入库流程。
- 原文件缺失：状态收敛为 `failed`，写入明确失败原因。
- 恢复以 `documents` 表作为持久事实来源，不依赖当前尚无消费者的任务队列 `pending` 状态。

### 3.4 上传交互一致性

- 当前前端明确采用单文件上传，移除 `multiple`。
- 拖放多个文件时只接收第一个，避免其余文件被静默拒绝或覆盖进度。

## 4. 验收场景

1. 上传前发起的空列表请求晚返回，不覆盖新文档。
2. 最新列表确认活动文档不存在，上传区域恢复可点击。
3. 处理中文档不能删除或清空。
4. 服务在解析/Embedding 中退出，重启后任务重新调度。
5. 恢复时原文件丢失，文档显示失败而非永久排队。
6. 清空成功后文档、上传锁、SSE 和重试状态全部清零。
