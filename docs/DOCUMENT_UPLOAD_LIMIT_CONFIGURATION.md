# 文档上传容量配置优化

> 日期：2026-07-16  
> 状态：已执行

## 目标

将知识库文件上限从前后端分别硬编码的 50 MB，升级为服务端统一管理的运行时配置：默认满足企业文档需求，同时保留明确的资源安全边界。

## 配置策略

- `DOCUMENT_MAX_UPLOAD_MB` 默认 `200`。
- 合法范围为 `1-512` MB。
- 超过 512 MB 时后端拒绝启动，避免错误配置绕过应用安全边界。
- 后端仍按 1 MB 流式落盘，提高容量不会线性提高上传内存。
- 前端通过 `/api/documents/upload-config` 获取真实限制，不再维护独立常量。
- 前端选择文件后立即校验大小，超限文件不会发起网络请求。

## 部署层配套

应用本身不包含 Nginx/Caddy。若部署时增加反向代理，其请求体限制必须不小于应用配置，并预留 multipart 开销。例如应用配置 200 MB 时：

```nginx
client_max_body_size 201m;
proxy_read_timeout 300s;
proxy_send_timeout 300s;
```

Caddy 默认不主动限制请求体；如果使用 `request_body max_size`，应采用同样的容量策略。

## 资源边界

文件字节数不是唯一成本。超大扫描 PDF 还会增加 OCR、文本切分、Embedding 和索引耗时。因此 512 MB 作为安全硬上限，不代表所有 512 MB 文档都适合直接入库。跨公网或更大文件应采用分片上传、断点续传和异步预处理。

## 验收标准

- 默认接口返回 200 MB。
- `.env` 和 Docker 可以覆盖默认值。
- 513 MB 配置无法通过 Settings 校验。
- 前端显示服务端返回的限制。
- 超限文件在浏览器侧拦截，服务端仍保留 HTTP 413 兜底。
