# 文档入库可靠性 V2 优化方案

> 日期：2026-07-16  
> 范围：知识库上传、重新处理、进度同步与故障恢复

## 1. 问题与目标

上一轮已解决“重试接口同步阻塞”和“点击后界面不刷新”。本轮继续处理长耗时任务中的连接可靠性：文档解析或 Embedding 超过 30 秒无阶段事件时，服务端会发送 `timeout` 并关闭 SSE；前端又会把网络错误视为任务结束，导致进度通道过早终止。

本轮目标：

- 长任务没有阶段事件时，进度连接保持可用。
- 短暂断网、代理重置或后端连接关闭后自动重连。
- 即使进程内进度事件丢失，也能从数据库恢复真实终态。
- 页面刷新或轮询发现终态后，清理本地“处理中”状态。

## 2. 优化设计

### 2.1 服务端：心跳与持久状态补偿

- SSE 每 15 秒发送注释心跳 `: keepalive`，不再把空闲误判为超时。
- 连接建立时先读取数据库并返回当前状态；若已成功或失败，立即结束连接。
- 每次空闲心跳前查询数据库。状态发生变化或进入终态时，补发标准进度事件。
- 下发 `retry: 3000`，建议浏览器在断线 3 秒后重连。

### 2.2 前端：以业务终态结束订阅

- `EventSource.onerror` 不主动关闭连接，交给浏览器执行标准自动重连。
- 仅 `ready`、`failed`、`not_found` 可以结束订阅。
- 终态处理增加幂等保护，避免重复事件触发多次刷新和 Toast。

### 2.3 状态收敛

- 文档列表刷新时，以服务端状态为准。
- 已进入终态或已删除的文档，自动清理本地 `reprocessing` 标记和残留订阅。
- 保留列表轮询作为 SSE 不可用时的第二通道。

## 3. 验收场景

1. Embedding 超过 30 秒：连接持续存在，不出现伪失败或伪完成。
2. SSE 中途触发网络错误：不调用完成回调，浏览器继续重连。
3. 完成事件丢失：下一次心跳从数据库读到 `ready/failed` 并补发。
4. 打开一个已失败文档的进度地址：立即收到失败原因并正常结束。
5. 页面轮询先观察到终态：重试按钮不会永久处于禁用状态。
