# 文档流式上传优化方案

> 日期：2026-07-16  
> 状态：已执行

## 1. 优化目标

原上传接口通过 `await file.read()` 一次性读取完整文件。对于大文件，请求体、Python `bytes` 和保存过程可能形成多份内存副本；前端同时无法获取真实上传百分比，也不能取消传输。

本轮目标：

- 服务端上传内存复杂度由 `O(文件大小)` 降为 `O(1 MB)`。
- 文件接收过程中增量计算 SHA-256，不增加第二次整文件扫描。
- 超限、空文件、连接异常和数据库冲突时清理临时文件。
- 上传完成后通过同文件系统原子移动进入正式目录。
- 前端展示真实传输百分比，并允许在上传阶段取消。

## 2. 实现方案

### 服务端

1. 每次读取 1 MB，写入上传目录中的 `.upload-*.tmp`。
2. 每个分块同时更新 SHA-256 和累计大小。
3. 超过运行时配置上限返回 HTTP 413，空文件返回 HTTP 400。
4. 完整接收后调用 `ingest_document_from_path()`：检查内容哈希、原子重命名、创建文档记录并提交后台处理。
5. 保留原 `ingest_document(bytes)`，兼容评测脚本、Worker 和已有测试。

### 前端

1. 文件上传使用 `XMLHttpRequest.upload.onprogress` 获取真实字节进度。
2. 使用 `AbortController` 连接“取消上传”按钮。
3. 上传阶段与解析、切分、向量化、索引阶段分开显示。
4. 同一时间只允许一个前台上传，避免全局进度状态相互覆盖。

## 3. 验收标准

- 读取调用的单块大小不超过 1 MB。
- 1 MB 以上文件的落盘内容、哈希和大小完全一致。
- 超限文件不留下 `.tmp` 文件或数据库记录。
- 上传进度能够从 0% 更新到 100%。
- 点击取消会中止网络请求并恢复上传区域。
- 文档完成上传后仍沿用现有 SSE 心跳、自动重连与后台入库流程。

## 4. 后续方向

对于跨公网的大文件场景，可继续增加分片上传和断点续传；对于当前单机/内网部署，本轮流式接收、取消和故障清理已经覆盖主要资源与交互风险。
