# RAG 检索精准度评测报告

> 评测日期：2026-07-06
> 评测范围：检索管线（hybrid_search），10 个查询用例 × 3 篇测试文档
> 环境：Qwen text-embedding-v2 / DeepSeek v4-flash / Qdrant local / bge-reranker-v2-m3

## 一、评测方法

### 1.1 测试语料

3 篇自建测试文档，覆盖不同领域：

| 文档 | 内容 | Chunks | Token 估算 |
|------|------|--------|------------|
| `ml_guide.txt` | Python 机器学习完全指南（框架、预处理、深度学习、模型评估） | 1 | ~500 |
| `climate_report.txt` | 全球气候变化报告（温度、温室气体、可再生能源、应对措施） | 2 | ~600 |
| `mediterranean_cooking.txt` | 地中海美食烹饪（Carbonara、希腊沙拉、西班牙海鲜饭） | 2 | ~600 |

### 1.2 查询用例（10 个）

每个查询标注了预期的相关 chunk，覆盖 3 篇文档的所有主题段落。

### 1.3 评测指标

| 指标 | 定义 | 说明 |
|------|------|------|
| **Precision@k** | 前 k 个结果中相关结果占比 | 精确率，越高越好 |
| **Recall@k** | 前 k 个结果召回的相关结果 / 总相关数 | 召回率，越高越好 |
| **MRR** | 首个相关结果排名的倒数平均值 | Mean Reciprocal Rank |
| **NDCG@k** | 位置加权相关性，DCG / IDCG | 考虑排序质量 |
| **Hit Rate@k** | 前 k 中至少命中 1 个相关结果的查询比例 | 覆盖能力 |

相关性判定采用词级 Jaccard 相似度（阈值 ≥0.25 或短文本词覆盖率 ≥50%）。

### 1.4 评测脚本

`backend/tests/evaluate_rag.py` — 独立运行，自动完成文档摄入 → 检索评测 → 指标计算 → 清理。

```bash
cd backend && python tests/evaluate_rag.py
```

## 四、检索来源分析

```
检索来源分布（无重排序）:
  语义检索:     100 (100%)  ← 完全主导
  关键词检索:     0 (0%)    ← 完全失效
  双源融合:       0 (0%)    ← RRF 融合无输入
```

**FTS5 关键词检索贡献为 0**。在 10 个查询 × 10 个结果的 100 条检索结果中，没有一条来自关键词搜索，没有一条是 RRF 双源融合的结果。

### 根因分析

`textdb/sqlite_fts.py` 中 `_escape_fts5()` 方法将整个查询包裹在双引号中做精确短语匹配：

```python
@staticmethod
def _escape_fts5(query: str) -> str:
    cleaned = re.sub(r'["*^()]', "", query)
    return '"' + cleaned.replace('"', '""') + '"'
```

而 `_segment_cjk()` 在存储时将 CJK 字符间插入空格（用于 FTS5 unicode61 分词器）。两相结合导致：

- **中文查询** "机器学习" → MATCH `" 机 器 学 习 "` → 带空格的精确短语，几乎不可能匹配
- **英文查询** "Carbonara" → MATCH `"Carbonara"` → 精确词匹配，但实际文档文本中该词前后可能有其他标点/字符
- **混合查询** "Python 深度学习" → MATCH `"Python  深 度 学 习 "` → 带空格的短语匹配，更不可能命中

**结果**：无论中英文查询，FTS5 MATCH 几乎都返回空集。语义搜索成了唯一的检索来源。

## 五、重排序效果分析

| 指标 | 无重排序 | 含重排序 | 变化 |
|------|---------|---------|------|
| Precision@5 | 22.00% | 20.00% | ↓ 9.1% |
| MRR | 85.00% | 87.00% | ↑ 2.4% |
| Hit Rate@5 | 90.00% | 100.00% | ↑ 10.0% |
| 延迟 | 291ms | 947ms | ↑ 656ms (3.3×) |

**评价**：在当前 512 token 大 chunk 配置下，重排序的边际收益很低。MRR 微涨 2.4% 但 P@5 反而下降，且延迟增加了 3.3 倍。大 chunk 已经包含了足够的语义密度，cross-encoder 难以在 chunk 内部进行细粒度区分。

## 六、发现的问题

### 6.1 P0 — FTS5 关键词检索失效

**影响**：混合检索退化为纯语义搜索，失去关键词精确匹配能力。对于代码 ID、型号（如 "XYZ-9000"）、专有名词等场景，语义搜索的效果远不如关键词匹配。

**修复方向**：
1. 修改 `_escape_fts5()`，对中文查询使用分词后的 OR 连接：`机 OR 器 OR 学 OR 习`
2. 或移除双引号包裹，让 FTS5 的 unicode61 tokenizer 自行处理中文分词
3. 考虑引入 jieba 等中文分词库，进行更精确的 CJK 分词

### 6.2 P1 — 大 Chunk 导致粒度粗糙

**影响**：512 token chunk 将整篇 ML Guide（500 token）合并为 1 个 chunk，4 个不同方向的查询全部映射到同一 chunk，无法区分排序质量。

**修复方向**：减小 `chunk_size` 至 256-384，或使用语义分块（按段落/标题边界分割）。

### 6.3 P1 — 数据污染

**影响**：Qdrant 中存在之前测试残留的旧数据（MCP、Skills、OpenCode 等主题），干扰检索结果排序。

**修复方向**：评测脚本中增加 Qdrant collection 和 FTS5 表的全量清空步骤。

### 6.4 P2 — 查询 9 "希腊沙拉" 完全未命中

**分析**：mediterranean_cooking.txt 的 chunk 1 同时包含"希腊沙拉"和"西班牙海鲜饭"两个主题。语义向量可能偏向 paella 相关内容，"希腊沙拉"的语义距离较远。

**修复方向**：减小 chunk size 将两个主题拆分到独立 chunk。

### 6.5 P2 — 重排序性价比低

**分析**：在当前配置下（大 chunk + Qwen embedding），reranker 带来的排序改善微乎其微，却增加了 3 倍延迟。

**修复方向**：减小 chunk size 后重排序的收益会更明显。也可考虑默认关闭 reranker，由用户按需开启。

## 七、优化建议优先级

| 优先级 | 问题 | 方案 | 预期效果 |
|--------|------|------|----------|
| **P0** | FTS5 关键词搜索失效 | 修改 `_escape_fts5()` 查询构建逻辑 | P@5 预计提升至 35-50%，关键词+语义双源生效 |
| **P1** | Chunk 大小 512 过大 | 降低至 256-384，或使用语义分割 | 提高 Precision，降低无关内容混入 |
| **P1** | 数据污染 | 评测前全量清空 Qdrant + FTS5 | 评测结果更准确 |
| **P2** | 重排序默认开启但收益低 | 默认关闭或按文档数动态开启 | 节省延迟，在小 chunk 下按需开启 |
| **P2** | 评测用例不足 | 扩展到 50+ 查询，覆盖更多边界情况 | 统计显著性更好 |

## 九、与 Agent 评测的关系

本报告评测的是 **检索管线层面**（retrieval precision），而 `docs/AGENT_EVALUATION.md` 评测的是 **Agent 子系统层面**（意图分类、记忆拦截、ReAct 循环）。两者互补：

| 维度 | 本报告 | Agent 评测 |
|------|--------|-----------|
| 评测对象 | `rag/retriever.py` hybrid_search | `agent/loop.py` + classifier + intercept |
| 核心指标 | Precision@k, MRR, NDCG | 意图分类准确率、重试机制正确性 |
| 发现的主要问题 | FTS5 关键词失效 | 工具重试死代码（已修复） |
| 综合评分 | C+ | 8.0/10（Agent 子系统） |

**结论**：Agent 框架设计良好，但检索基座需要修复 FTS5 关键词搜索和 Chunk 粒度问题才能发挥全部潜力。
