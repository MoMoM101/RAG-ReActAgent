# 项目优化第九阶段：Markdown 最终答案规范化（2026-07-21）

## 结论

用户以真实页面 DOM 复测“skill 是什么”及追问“和 mcp 有什么区别”后，确认前端 Markdown 解析器工作正常，但后端最终文本存在结构损坏：搜索过程话术与标题粘连，grounding 修复把表格行包装成列表项，导致浏览器只能渲染成普通段落或包含竖线文本的列表。

本阶段将 Markdown 保障从提示词要求升级为“语义补全、完整答案缓冲、确定性格式规范化、缓存失效和前端 DOM 清理”五层约束。没有调用 Claude 或外部模型 API。

## 1. 真实问题

附件中的实际输出包含：

```text
在知识库中搜索一下相关定义。### Skill 是什么
```

由于 `###` 不在行首，它不是 Markdown 标题。

比较回答包含连续的以下结构：

```text
- | 对比维度 | MCP | Agent Skills | [S1, S2]。
- | 核心价值 | 协议层标准化 | 应用层封装 | [S4]。
```

由于每个表格行前存在列表标记且缺少分隔行，GFM 会把它解析为普通列表，不会生成表格。

此外，`和mcp有什么区别` 省略了比较左侧对象。旧逻辑依赖模型从历史猜测 Skill，确定性完整性守卫无法验证回答是否同时覆盖双方。

## 2. 修复

- 新增最终答案 Markdown 规范化器；
- 删除确定属于过程说明的“搜索一下、检索一下”式开场；
- 将粘在句子后的 Markdown 标题移动到独立行；
- 修复结束标记前多余空格造成的粗体失效；
- 将连续 `- |...|` 行恢复为真正的 GFM 表格，并自动生成列数一致的分隔行；
- fenced code 内的 Markdown 示例保持原样，不参与改写；
- 定义、概览、详细说明和比较题等待完整生成后统一验证及规范化，避免已发送内容无法撤回；
- `和mcp有什么区别` 可根据上一轮“skill是什么”确定性补全为 `skill和mcp有什么区别`；
- 验证器忽略 Markdown 表头和分隔行，只校验有事实含义的数据行；
- 缓存版本提升为 `v7-markdown-normalized`，不复用旧的畸形格式答案；
- 清除 `react-markdown` 内部 `node` 对象被透传到 `<code>` 和链接 DOM 属性的问题。
