# 项目优化第八阶段：Markdown 输出与渲染（2026-07-21）

## 结论

本阶段修复了“前端回答看起来不是 Markdown”的实际体验问题。审计确认原前端已经使用 `ReactMarkdown + remark-gfm`，根因主要是后端输出契约允许普通单段文本，因此解析后的视觉效果与纯文本相同；同时存在宽表格缺少横向滚动容器、无语言代码围栏被当作行内代码的渲染边界。

优化后，知识回答会按题型输出稳定的 GitHub Flavored Markdown 结构，前端完整支持标题、列表、表格、链接、引用、分隔线、行内代码及代码块。Agent、前端测试、静态检查、生产构建和发布门禁全部通过。本阶段没有调用 Claude 或外部模型 API。

## 1. 后端输出契约

系统提示词新增以下约束：

- 知识回答输出可直接渲染的 GitHub Flavored Markdown；
- 不把整段答案放入 `markdown` 代码围栏；
- 单一事实使用加粗“结论”标签；
- 定义、概览和详细说明使用三级标题与要点列表；
- 步骤和操作方法使用有序列表；
- 比较题使用双方三级标题及各自要点，必要时单列无法确认部分；
- 标题只组织已有证据，不得为了格式补充无来源事实；
- 禁止空标题、空列表及只有标题没有正文的答案。

同样的 Markdown 约束已同步到完整答案修复和流式后半段修复提示，避免初稿格式正确、修复后又退化成普通段落。

## 2. 前端渲染

消息组件新增或完善：

- 标题层级的字号、间距与颜色；
- 列表标记及嵌套段落间距；
- 表格外层横向滚动容器，避免窄屏撑破消息区域；
- 表格边框与单元格分隔；
- 引用块首尾间距和分隔线样式；
- Markdown 链接在新标签页打开，并设置 `noopener noreferrer`；
- 无语言 fenced code 正确渲染为带复制按钮的代码块，而不是行内代码。

用户输入仍按纯文本显示；Markdown 渲染只应用于助手回答，避免用户输入被意外解释成富文本。
